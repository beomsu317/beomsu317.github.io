Dagger

Dagger는 DI Framework 중 하나로 리플렉션을 사용하지 않고 컴파일 타임에 코드를 생성해 빠르게 동작한다. 컴파일 타임에 코드를 생성할 수 있는 이유는 Annotation Processing 덕분이다.

@Module은 클래스에만 붙이며 @Provide는 @Module 클래스 안의 선언된 메서드에만 붙인다. Module Class의 의존성 주입에 필요한 객체들은 Provide 메서드를 통해 관리한다. Provide 메서드의 파라미터 또한 컴포넌트 구현체로부터 전달받을 수 있다.

Dagger에는 5가지 개념이 존재한다.

  • Inject
  • Module
  • Component
  • SubComponent
  • Scope

Inject

의존성 주입을 요청한다. Inject Annotation으로 주입을 요청하면 연결된 Component가 Module로부터 객체를 생성하여 넘겨준다.

Dagger가 클래스의 인스턴스를 만들 생성자에 @Inject 어노테이션을 사용한다. 새로운 인스턴스 생성이 요청되면, Dagger는 파라미터와 값들을 얻고 이 생성자를 호출한다.

class Thermosiphon implements Pump {
  private final Heater heater;

  // 생성자에 의존성 주입
  @Inject
  Thermosiphon(Heater heater) {
    this.heater = heater;
  }
}

Dagger는 필드를 직접적으로 Inject 할 수 있다. 다음은 Heater와 Pump 인스턴스를 얻는 예제이다.

1
2
3
4
5

class CoffeeMaker {
  // 필드에 의존성 주입
  @Inject Heater heater;
  @Inject Pump pump;
}

클래스에 @Inject-annotated 필드가 있지만, @Inject-annotated 생성자가 없을 경우, Dagger는 요청된 필드를 Inject 하지만 새로운 인스턴스를 만들지 않는다.

Dagger는 또한 메서드 Injection을 지원하지만 전형적으로 생성자나 필드 Injection을 사용한다.

Module

Module은 객체를 주입하기 위한 메서드를 정의한다. Component에 연결되어 의존성 객체를 생성한다. 생성 후 Scope에 따라 관리한다. Module Class는 클래스 이름 뒤에 Module을 붙이고, Provide 메서드명 앞에는 provide를 붙이는 것이 일반적인 컨벤션이다.

@Module
class CoffeeMakerModule {
    // 객체를 Component에게 제공
    @Provides
    fun provideHeater() : Heater = A_Heater()

    // 필요한 인자를 Component로부터 전달받아 Pump 객체를 생성해 Component에게 제공
    @Provides
    fun providePump(heater: Heater) : Pump = A_Pump(heater)
}

Dagger는 기본적으로 Null 인젝션을 금지하고 있다. Null을 인젝션하고 싶다면 @Nullable 어노테이션을 @Provide 메서드와 객체 주입받을 타입에 모두 사용할 경우에만 허용된다.

Component

연결된 모듈을 이용하여 의존성 객체를 생성하고, Inject로 요청받은 인스턴스에 생성한 객체를 주입한다. Interface 또는 abstract class에만 사용할 수 있다. 컴파일 타임에 Annotation Processor에 의해 생성된 클래스는 접두어 Dagger와 @Component가 붙은 클래스이름이 합쳐진 형식의 이름을 갖는다.

Component Methods

컴포넌트 메서드에는 Provision과 Member-Injection이 있다. Provision 메서드는 매개변수가 없고, 모듈이 제공하는 객체의 타입을 반환형으로 갖는다. 생성된 컴포넌트 클래스에서 이 메서드를 이용해 객체를 얻을 수 있다. Member-Injection 메서드는 의존성을 주입시킬 객체를 파라미터로 넘기는 방법이다. 클래스 내 @Inject 필드에 객체를 주입받는다.

@Component(modules = {AModule.class, BModule.class})
public interface MyComponent {
    //provision 메소드
    AA makeAA();

    //member-injection 메소드
    void inject(Main target);
}

@Component.Builder

@Component.Builder는 컴포넌트를 생성하기 위한 Builder용 Annotation이다. Component 내 abstract 클래스 또는 interface에 이 어노테이션을 붙인다. Builder는 반드시 Component를 반환하는 메서드와 Builder를 반환하면서 컴포넌트가 필요로 하는 모듈을 파라미터로 받는 메서드를 가지고 있어야 한다. @Component.Builder를 지정하지 않으면 암시적으로 Builder가 생성된다.

@Component(modules = [Moduel_A::class, Module_B::class])
interface MyComponent {

    @Component.Builder
    interface Builder {

        fun moduleA(moduleA: Module_A) : Builder

        fun moduleB(moduleB: Module_B) : Builder

        fun build() : MyComponent
}

Builder는 반드시 Component를 반환하는 메서드(여기선 build())와 Builder를 반환하면서 Component가 필요로하는 Module을 파라미터로 받는 메서드(여기선 moduleA())를 가지고 있어야 한다.

Builder를 반환하며 Module을 파라미터로 받는 메서드는 무조건 1개의 파라미터만 받는 추상 메서드로 구현해야 한다. 파리미터를 2개 이상 선언할 수 없기 떄문에, 여러 Module을 초기화하려면 각 Module마다 메서드를 추가해 체이닝 방식으로 Component 인스턴스를 생성해야 한다.

@Component.Factory

Builder와 의미는 동일하지만 사용방법이 약간 다르다. Builder는 파라미터를 1개만 받으며 여러 Module을 설정해야 한다면 체이닝 방식으로 구현해야 한다. Factory는 이 부분을 보완한 것이며 단 하나의 메서드(create())만 선언되어야 하고 반환 타입은 Component 인스턴스여야 한다. create() 메서드 하나만으로 설정이 가능하기 때문에 코드가 간결해진다.

@Component(modules = [Moduel_A::class, Module_B::class])
interface MyComponent {

    @Component.Factory
    interface Factory {

        fun create(moduleA: Module_A, moduleB: Module_B) : MyComponent
}

// Usage
DaggerMyComponent.factory()

Example

Module

먼저 의존성 관계를 설정하는 클래스(Module)를 정의한다. @Module 어노테이션을 선언한다.

Module 클래스에서는 객체를 주입하기 위한 메서드를 정의한다. 이 메서드들은 @Provides 어노테이션을 선언한다. Heater와 Pump 인터페이스 구현체를 주입하기 위한 메서드를 구현한다.

class CoffeeMakerModule {
    @Provides
    fun provideHeater() : Heater = A_Heater()

    @Provides
    fun providePump(heater: Heater) : Pump = A_Pump(heater)
}

Component

Component는 interface또는 abstract class로 선언해야 하며, 최소 1개 이상의 abstract method가 포함되어야 한다. abstract method를 통해 주입이 가능하다.

abstract method는 2가지 유형으로 구분된다.

  1. Provision Return Method : Injection 시킬 객체를 넘기는 유형
  2. Members-injection Method : 멤버 파라미터에 의존성 주입 시킬 객체를 넘기는 유형

@Component 어노테이션에 속성으로 modules를 설정했는데 이는 해당 Compnent에 의존성 주입을 구현할 모듈들을 알려달라는 의미이다. 그러면 Dagger는 CoffeeMakerModule에 @Provides로 구현된 메서드들을 이용해 의존성을 주입시킬 코드를 자동으로 생성해준다.

@Component(modules = [CoffeeMakerModule::class])
interface CoffeeComponent {

    // provision method 
    fun make() : CoffeeMaker

    // member-injection method 
    fun inject(coffeeMaker: CoffeeMaker)
}

Inject

@Inject를 통해 의존성을 주입한다. 생성자 또는 필드에 @Inject 어노테이션을 선언해 사용한다.

우선 Provision method인 make()를 사용한다.

CoffeeMaker 클래스의 주 생성자에 @Inject를 선언해 의존성 주입 대상임을 Dagger에게 알려준다. 주 생성자는 반드시 반환 타입이 CoffeeMaker이기 때문에 Component에서 반환타입이 같은 메서드를 찾아 주입해준다. Dagger는 Build 시 @Component가 달린 인터페이스를 Dagger 클래스로 자동으로 구현해준다. 구현되는 Dagger 클래스명은 선언된 Componenr 인터페이스 이름 앞에 Dagger 접두어를 붙여 생성된다. 여기선 인터페이스명이 CoffeeComponent이며 Dagger 클래스는 DaggerCoffeeComponent가 생성된다.

class CoffeeMaker @Inject constructor(private val heater: Heater, private val pump: Pump) {
    fun brew() {
        heater.on()
        pump.pump()
        Log.d("coffeMaker", "[_]P coffee! [_]P")
        heater.off()
    }
}

class CoffeeMaker() {

Member-injection method는 inject() 메서드에 파라미터로 전달받은 coffeeMaker 객체의 내부에 @Inject 어노테이션이 붙은 필드가 있다면 Module을 통해 의존성 주입을 실행하는 메서드이다.

DaggerCoffeeComponent.create().inject(coffeeMaker)
coffeeMaker.brew()

// MainActivity에서 의존성 주입 - 사용
class MainActivity : AppCompatActivity() {

    // ActivityMain의 멤버변수(필드)
    @Inject
    lateinit var heater: Heater

    // ActivityMain의 멤버변수(필드)
    @Inject
    lateinit var pump: Pump

    override fun onCreate(savedInstanceState: Bundle?) {
        super.onCreate(savedInstanceState)
        setContentView(R.layout.activity_main)

        // inject() 함수로 MainActivity의 멤버변수에 의존성을 주입
        DaggerCoffeeComponent.create().inject(this@MainActivity)

        var coffeeMaker = CoffeeMaker(heater, pump)	// 의존성 주입된 heater와 pump 사용
        coffeeMaker.brew()
    }
}
